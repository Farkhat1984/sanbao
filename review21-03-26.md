## 1. DRY-нарушения

### 1.1 CRITICAL: Admin CRUD Factory используется в 3 из 14 роутов

Фабрика `src/lib/admin-crud-factory.ts` существует, но **11 admin `[id]` роутов** дублируют логику вручную:

| Роут | Проблема |
|------|----------|
| `api/admin/providers/[id]/route.ts` | Ручной GET/PUT/DELETE, allowedFields loop |
| `api/admin/tools/[id]/route.ts` | Ручной PUT/DELETE с allowedFields |
| `api/admin/skills/[id]/route.ts` | Ручной GET/PUT/DELETE |
| `api/admin/users/[id]/route.ts` | Ручной PUT/DELETE |
| `api/admin/plans/[id]/route.ts` | Ручной PUT без DELETE |
| `api/admin/models/[id]/route.ts` | Ручной GET/PUT/DELETE с транзакциями |
| `api/admin/mcp/[id]/route.ts` | Ручной PUT/DELETE |
| `api/admin/system-agents/[id]/route.ts` | Ручной GET/PUT/DELETE (176 строк) |
| + ещё 3 роута | — |

**~400+ строк** дублированного кода. Пример ручного роута vs. фабрики:

```typescript
// ❌ РУЧНОЙ (providers/[id]/route.ts, 35 строк)
export async function PUT(req, { params }) {
  const result = await requireAdmin();
  if (result.error) return result.error;
  const { id } = await params;
  const body = await req.json();
  const provider = await prisma.aiProvider.findUnique({ where: { id } });
  if (!provider) return jsonError("Провайдер не найден", 404);
  const allowedFields = ["name", "slug", "baseUrl", "apiKey", "isActive", "priority"];
  const data = {};
  for (const field of allowedFields) {
    if (body[field] !== undefined) data[field] = body[field];
  }
  const updated = await prisma.aiProvider.update({ where: { id }, data });
  return jsonOk(updated);
}

// ✅ ФАБРИКА (experiments/[id]/route.ts, 6 строк)
export const { PUT, DELETE } = createAdminCrudHandlers({
  model: "promptExperiment",
  allowedUpdateFields: ["name", "description", "promptA"],
  notFoundMsg: "Эксперимент не найден",
  afterUpdate: invalidateExperimentCache,
});
```

### 1.2 HIGH: Auth-guard дублирован 130+ раз

```typescript
// Повторяется в КАЖДОМ API роуте:
const result = await requireAuth();
if ("error" in result) return result.error;
const { userId } = result.auth;
```

**Решение:** Next.js middleware или обёртка-декоратор `withAuth(handler)`.

### 1.3 HIGH: Пагинация — 3 разных реализации в 9+ роутах

| Тип | Метод | Роуты |
|-----|-------|-------|
| A — pop | `items.pop(); hasMore = items.length > limit` | agents, conversations |
| B — slice | `skills.slice(0, limit)` | admin/agents, skills |
| C — offset | `parsePagination(searchParams); skip = (page-1)*limit` | admin/users, admin/skills |

Нет единой утилиты пагинации. Курсорная и offset-пагинация смешаны произвольно.

### 1.4 MEDIUM: JSON-парсинг ошибок — 33 роута

```typescript
const body = await req.json().catch(() => null);
if (!body) return jsonError("Неверный JSON", 400);
```

### 1.5 MEDIUM: Date-сериализация — 43 ручных `.toISOString()` при наличии `serializeDates()`

Хелпер `serializeDates()` в `api-helpers.ts` существует, но используется не везде.

### 1.6 MEDIUM: Ассоциации (many-to-many) — `deleteMany + createMany` копипаста в 11+ роутах

```typescript
// Один и тот же паттерн для skills, tools, mcpServers:
await tx.agentSkill.deleteMany({ where: { agentId: id } });
await tx.agentSkill.createMany({
  data: skillIds.map(skillId => ({ agentId: id, skillId })),
});
```

**Решение:** `updateAssociations(tx, entityId, { skillIds, toolIds, mcpServerIds })`.

### 1.7 MEDIUM: Zustand-сторы — идентичный CRUD в 15 сторах

`agentStore`, `skillStore`, `taskStore` и др. содержат одинаковые `add/update/remove` методы. Нужна generic фабрика.

### 1.8 LOW: Select/Include паттерны для Agent определены в 3+ местах

`AGENT_SELECT`, `AGENT_INCLUDE`, `SYSTEM_AGENT_INCLUDE` — похожие структуры в разных файлах.

**Итого дублированного кода: ~1,900+ строк**

## 2. Модульность и архитектура

### 2.1 CRITICAL: Нет Service/Repository слоя

**Все 120+ API роутов** импортируют `prisma` напрямую. Нет абстракции между роутами и базой данных.

```
Текущее:  Route → Prisma (напрямую)
Нужно:    Route → Service → Repository → Prisma
```

Проблемы:
- Изменение схемы БД требует правок во всех роутах
- Невозможно unit-тестировать бизнес-логику
- Нет переиспользования запросов

### 2.2 CRITICAL: God-файлы

| Файл | Строк | Проблема |
|------|-------|----------|
| `src/lib/settings-registry.ts` | 1,608 | 118 настроек, UI-мета, типы, валидация в одном файле |
| `src/lib/chat/moonshot-stream.ts` | 840 | SSE-парсинг + MCP вызовы + native tools + форматирование |
| `src/lib/tool-resolver.ts` | 345 | Кэширование + бизнес-логика + DB-запросы |
| `src/lib/email.ts` | 320 | Транспорт + отправка + верификация + 6 шаблонов |
| `src/app/api/chat/route.ts` | 323 | Оркестрация + multi-agent + usage tracking |

### 2.3 HIGH: Бизнес-логика в роутах

`api/chat/route.ts` содержит:
- `handleMultiAgentMode()` с Prisma-запросами (строки 255-279)
- Multi-agent dispatch (строки 61-78)
- Usage tracking (строки 145-162)

Всё это — service layer, не route layer.

### 2.4 HIGH: Context Loader — 4 ответственности в одном файле

`api/chat/context-loader.ts` (268 строк):
1. Загрузка контекста из БД
2. LLM-компрессия (HTTP к провайдерам)
3. Redis lock management
4. Token usage tracking

### 2.5 HIGH: 16 импортов в chat/route.ts

Route-файл импортирует из 10+ модулей — нарушение Facade pattern.

### 2.6 MEDIUM: chatStore.ts — streaming логика в store

Streaming phase logic (строки 274-290) с вложенными условиями не должна быть в Zustand store.

### 2.7 MEDIUM: resetStores.ts — ручной сброс 15 сторов

Хрупкий паттерн: новый стор → ручное добавление reset-логики. Нужен registry pattern.

## 3. Паттерны проектирования

### 3.1 ANTI-PATTERN: Strategy вместо if-else для AI-провайдеров

```typescript
// ❌ Текущее (route.ts):
if (apiFormat === "OPENAI_COMPAT") {
  const stream = streamMoonshot(...);
} else {
  const stream = await streamAiSdk(...);
}

// ✅ Нужно:
interface StreamProviderStrategy {
  canHandle(format: ApiFormat): boolean;
  stream(messages, options): ReadableStream;
}
```

Добавление нового провайдера требует модификации route.ts.

### 3.2 ANTI-PATTERN: Нет Middleware паттерна

`src/middleware.ts` отсутствует. Каждый роут самостоятельно:
- Проверяет авторизацию
- Устанавливает correlation ID
- Проверяет rate limiting

### 3.3 ANTI-PATTERN: Promise-based таймауты вместо AbortController

```typescript
// ❌ mcp-client.ts (устаревший подход):
function withTimeout<T>(promise: Promise<T>, ms: number): Promise<T> {
  return new Promise((resolve, reject) => {
    const timer = setTimeout(() => reject(...), ms);
    promise.then(v => { clearTimeout(timer); resolve(v); }, ...);
  });
}

// ✅ Современный:
const controller = new AbortController();
setTimeout(() => controller.abort(), ms);
```

### 3.4 GOOD: Discriminated Union для ошибок

```typescript
// Хороший паттерн — используется последовательно:
if ("error" in result) return result.error;
```

### 3.5 GOOD: Admin CRUD Factory

Отличная реализация (когда используется) — whitelist полей, callbacks, трансформеры.

### 3.6 GOOD: MCP Tool Deduplication

Двухпроходный алгоритм в `agent-resolver.ts` — корректно обрабатывает коллизии имён.

### 3.7 GOOD: Native Tools Registry

Паттерн plugin-архитектуры с `registerNativeTool()`.

### 3.8 MEDIUM: Широкое использование `any` типов

```typescript
// ai-sdk-stream.ts:
messages: any[];
fullStream: AsyncIterable<any>;
```

Нивелирует преимущества TypeScript. Нужны proper generics.

### 3.9 MEDIUM: Streaming state разбросан по полям

`streamingContent`, `streamingReasoning`, `streamingPlanContent`, `streamingPhase` — должен быть единый `StreamingState` объект.

## 4. Микросервисная архитектура

### 4.1 Docker Compose: 9/10

- 11 контейнеров с правильной изоляцией
- Цепочка зависимостей с health checks
- Resource limits установлены
- Security hardening (read-only fs, no-new-privileges)

### 4.2 CRITICAL: Единая база данных — нет domain isolation (1/10)

- 68 моделей Prisma в одной схеме
- Один `prisma.ts` используется всеми 134 роутами
- Нет per-service баз данных
- Удаление User каскадно удаляет ВСЁ (conversations, messages, tokens, artifacts)

### 4.3 HIGH: Монолит перегружен (4/10)

134 API роута, ~3MB кода. Кандидаты на извлечение:

| Домен | Роуты | Рекомендация |
|-------|-------|--------------|
| Admin | 29 | Отдельный admin-сервис |
| Billing | 6 | Payment adapter |
| Agents/Skills/Tools | 10+ | Knowledge & Tool Registry |
| Organizations | 10+ | Multi-tenant management |
| Chat | 1 + 4 deps | Core, но вынести в service layer |

### 4.4 HIGH: Нет Circuit Breaker для orchestrator (3/10)

```typescript
// ai-cortex-client.ts — один fetch, один throw:
const res = await fetch(`${AI_CORTEX_URL}${path}`, { signal });
if (!res.ok) throw new AiCortexError(...); // Нет retry, нет circuit breaker
```

Если orchestrator под нагрузкой → все `/api/chat` запросы висят 30с → fail.

### 4.5 HIGH: Вебхуки диспатчатся последовательно

```typescript
// webhook-dispatcher.ts:
for (const webhook of webhooks) {  // ❌ Последовательно
  ...retry loop с экспоненциальным backoff...
}
// 10 вебхуков × 1 таймаут = 300 секунд задержка
```

**Нужно:** `Promise.allSettled()` для параллельного dispatch.

### 4.6 GOOD: Nginx конфигурация (8/10)

- ip_hash sticky sessions (OAuth PKCE)
- Layered rate limiting (10r/s chat, 30r/s general)
- SSE-специфичные таймауты (180s, proxy_buffering off)
- Кэширование 1C-изображений (30d)
- Security headers (HSTS 2yr, X-Frame-Options DENY)

### 4.7 GOOD: Deploy автоматизация (9/10)

- Zero-downtime rolling restart
- Tmux wrapping для SSH
- Cloudflare cache purge
- Failover с Telegram-уведомлениями

### 4.8 GOOD: BullMQ с graceful degradation (8/10)

Redis доступен → очередь с retry. Redis недоступен → inline execution.

## 5. Готовность к продакшену

### 5.1 CRITICAL: Секреты в .env в репозитории

**`.env` содержит продакшн-ключи:**
- `MOONSHOT_API_KEY="sk-80eqv3Svq41i..."` — активный API ключ
- `DEEPINFRA_API_KEY="5O48pzde3nh..."` — открыт
- Google OAuth credentials (3 платформы)
- `AI_CORTEX_AUTH_TOKEN` — полный токен
- SMTP credentials (`zfaragj@gmail.com` + пароль)

**Действие:** Немедленно ротировать все ключи. Удалить `.env` из git-истории. Внедрить secrets management.

### 5.2 CRITICAL: CSP — `unsafe-inline` + `unsafe-eval`

```typescript
// proxy.ts:
"script-src 'self' 'unsafe-inline' 'unsafe-eval' ..."
```

Полностью обходит CSP. Любой XSS-вектор выполнит произвольный JavaScript.

### 5.3 CRITICAL: Admin fallback credentials

```typescript
// auth/login/route.ts:
const ADMIN_LOGIN = process.env.ADMIN_LOGIN || "admin";
```

Если `ADMIN_PASSWORD` не задан — возможен backdoor.

### 5.4 HIGH: Redis optional → rate limiting в multi-replica обходится

В продакшене с 3 репликами, каждая ведёт отдельные счётчики rate-limit. Атакующий распределяет запросы по репликам → лимит ×3.

### 5.5 HIGH: Тестовое покрытие — 6%

28 тест-файлов vs. 453 source-файлов. Отсутствуют:
- Integration тесты для `/api/chat`
- E2E тесты для billing
- Database migration тесты
- MCP integration тесты
- Rate limiting тесты

### 5.6 HIGH: ENCRYPTION_KEY валидация отсутствует

```typescript
// crypto.ts:
const secret = process.env.ENCRYPTION_KEY;
if (!secret) {
  const fallback = process.env.NEXTAUTH_SECRET || process.env.AUTH_SECRET;
  // Слабый fallback, нет проверки длины
}
```

### 5.7 MEDIUM: 41+ CASCADE DELETE без защиты

Удаление `User` каскадно удаляет: Conversations → Messages → Artifacts, Plans, LegalReferences, TokenLogs, Notifications, ApiKeys, OrgMemberships...

Одна ошибка в admin-панели = полная потеря данных пользователя без возможности восстановления.

### 5.8 MEDIUM: console.log/console.warn в продакшн-коде

```typescript
// instrumentation.ts:
console.log("[instrumentation] Workers and shutdown handlers registered");
// crypto.ts:
console.warn("[CRYPTO] Using NEXTAUTH_SECRET/AUTH_SECRET...");
```

Ломает structured JSON logging.

### 5.9 MEDIUM: Docker seed failures игнорируются

```bash
# docker-entrypoint.sh:
node prisma/seed.js 2>&1 || {
  echo "WARNING: seed failed, continuing..."
}
```

Отсутствие seed-данных (Plans, Agents, Tools) → runtime failures.

### 5.10 MEDIUM: Health check race при graceful shutdown

App получает SIGTERM → начинает drain → Docker healthcheck видит 503 → может убить pod до завершения drain.

### 5.11 LOW: CORS неполный для mobile-клиентов

CORS-заголовки только на OPTIONS preflight. GET/POST/PUT/DELETE ответы без CORS → Capacitor запросы ломаются.

### 5.12 LOW: SVG/Script XSS при загрузке файлов

Нет валидации что загруженные файлы безопасны для preview. SVG может содержать `<script>`.

## 6. Итоговая оценка

| Категория | Оценка | Комментарий |
|-----------|--------|-------------|
| **DRY** | 4/10 | ~1,900 строк дублирования, фабрика не используется |
| **Модульность** | 3/10 | Нет service/repository layer, god-файлы |
| **Паттерны** | 5/10 | Хорошие основы, но missing Strategy, Middleware |
| **Микросервисы** | 5.5/10 | Инфраструктура отличная, но DB coupling критичный |
| **Security** | 3/10 | Секреты в repo, unsafe CSP, weak encryption fallback |
| **Тестирование** | 2/10 | 6% покрытие, нет integration/E2E |
| **Production readiness** | 4/10 | Деплой хороший, но security и reliability gaps |

### **Общая оценка: 3.8/10 — НЕ ГОТОВ К ПРОДАКШЕНУ**

### Почему проект не готов:

1. **Безопасность:** Продакшн-секреты в git, CSP обходится, rate limiting обходится при multi-replica
2. **Надёжность:** Нет circuit breaker для критичных сервисов, cascade delete без защиты
3. **Тестирование:** 6% покрытие — нет уверенности что код работает корректно
4. **Архитектура:** Монолит без слоёв — невозможно масштабировать, тестировать, поддерживать
5. **DRY:** 1,900+ строк дублирования — баги фиксятся в одном месте, но остаются в 10 других
